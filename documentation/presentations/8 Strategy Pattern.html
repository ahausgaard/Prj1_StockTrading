Strategy Pattern

Indkapsling og udskiftning af algoritmer

Software Design og Test

Del A: Hvad er formålet med mønstret?

"Definer en familie af algoritmer, indkapsl dem, og gør dem udskiftelige."

Strategy Pattern er et adfærdsmønster, der lader dig ændre en klasses adfærd eller algoritme at runtime.

I stedet for at en klasse (Context) implementerer flere forskellige måder at gøre en ting på (f.eks. beregne fragt, sortere lister, eller komprimere filer), delegerer den ansvaret for hvordan opgaven løses til et selvstændigt strategi-objekt.

Del B: Hvordan ser det ud UDEN mønstret?

Ligesom med State-mønstret resulterer manglen på Strategy typisk i gigantiske switch-statements (Brud på Open/Closed Principle).

public class Indkøbskurv {
    public decimal BeregnFragt(string kurer, double vægt) {
        if (kurer == "GLS") {
            return 39.00m + (decimal)vægt * 2;
        }
        else if (kurer == "PostNord") {
            return 49.00m + (decimal)vægt * 3;
        }
        else if (kurer == "Afhentning") {
            return 0.00m;
        }
        throw new Exception("Ukendt kurer!");
    }
}

Del B: Implementering - Strategierne

Vi indkapsler hver beregningsalgoritme i sin egen klasse via et fælles interface.

// Interfacet: Definerer handlingen
public interface IFragtStrategi {
    decimal Beregn(double vægt);
}

// Konkrete strategier (Algoritmerne)
public class GlsStrategi : IFragtStrategi {
    public decimal Beregn(double vægt) => 39.00m + (decimal)vægt * 2;
}

public class PostNordStrategi : IFragtStrategi {
    public decimal Beregn(double vægt) => 49.00m + (decimal)vægt * 3;
}

Del B: Implementering - Context (Brugen)

Vores Indkøbskurv (Context) kender nu kun til interfacet. Den får "indsprøjtet" strategien udefra (Ofte via Dependency Injection).

public class Indkøbskurv {
    private IFragtStrategi _fragtStrategi;

    // Strategien kan sættes eller skiftes at runtime
    public void SætFragtStrategi(IFragtStrategi strategi) {
        _fragtStrategi = strategi;
    }

    public decimal UdregnTotal(double vægt) {
        // Kurven ved ikke HVORDAN fragten beregnes, den beder bare om det
        decimal fragtPris = _fragtStrategi.Beregn(vægt);
        return 100.00m + fragtPris; // Varepris + Fragt
    }
}

Hvor og hvordan er det brugt? (Del B)

Anvendelsesscenarier

  • Betalingsgateways: Brugeren vælger PayPalStrategy, CreditCardStrategy eller MobilePayStrategy ved udtjekning.
  • Sortering (LINQ i C#): .OrderBy(x => x.Navn). Den underliggende algoritme kan skiftes mellem QuickSort eller MergeSort baseret på listens størrelse.
  • Filkomprimering: Gem som .ZIP, .RAR, .TAR.

Testbarhed

Strategier er drømmen for en tester. Algoritmerne for at beregne PostNord-fragt kan unit-testes totalt uafhængigt af indkøbskurven. Samtidig kan indkøbskurven testes isoleret ved hjælp af en simpel MockFragtStrategi.

Klassisk spørgsmål: Strategy vs. State

Deres UML-diagrammer er næsten identiske! Men forskellen ligger i Intentionen:

State Pattern

Tilstandsobjektet skifter sig selv ud. Objektet reagerer på, hvad der sker med det (f.eks. Ordren bliver betalt, så State skifter koden til `BetaltTilstand`). Lukket økosystem.

Strategy Pattern

Strategiobjektet skifter aldrig sig selv. Det er Klienten (brugeren/systemet), der udefra bestemmer: "Brug denne strategi nu". Udskiftelige værktøjer.

Den kritiske vinkel: Faldgruben

Klienten skal kende implementeringerne.

For at kunne udskifte en strategi, er brugeren (f.eks. UI'et) ofte nødt til at kende til de forskellige strategier for at kunne give den rigtige videre til Context.

Dette overfører ofte logikken om *hvilken* strategi der skal vælges op i brugergrænsefladen, hvilket skaber en anden form for tæt kobling. Dette løses typisk ved at kombinere Strategy-mønstret med et Factory-mønster, der tager sig af at oprette den korrekte strategi.

Opsummering

  • Composition over Inheritance: Giver adfærd til en klasse via en variabel, i stedet for at skulle nedarve fra klassen for at ændre metoder.
  • Open/Closed Principle: Nye algoritmer kræver blot en ny klasse, uden at rette i eksisterende kode.
  • Klient-drevet: Det er brugeren eller systemet udefra, der bestemmer, hvilken "værktøjs-strategi" der er bedst til opgaven lige nu.
