MVVM Pattern

Model-View-ViewModel: Skab testbare brugergrænseflader

Software Design og Test

Hvorfor MVVM?

"Ingen Code-Behind. UI er dumt, logikken er smart."

MVVM er et arkitekturmønster primært brugt til brugergrænseflader (f.eks. WPF, .NET MAUI, Blazor og moderne Web Frameworks).

Formålet er adskillelse: At afkoble den grafiske brugerflade (View) totalt fra forretningslogikken og dataene (Model). Dette gøres ved at introducere en mellemmand (ViewModel).

Mønstrets UML-struktur

Den vigtigste regel i MVVM er afhængigheds-retningen. Viewet kender ViewModel. ViewModel kender Model. Men pilen peger aldrig tilbage!

View

XAML / HTML
Knapper, tekstfelter.

DataBinding → ← INotifyPropertyChanged Commands →

ViewModel

Præsentationslogik.
Holder tilstanden.

Opdaterer → ← Events/Data

Model

Forretningslogik.
Databaser, domæne.

Filosofien bag: DataBinding & Magi

Hvordan kommunikerer de, når ViewModel ikke må kende View'et (f.eks. ikke må skrive `txtNavn.Text = "Jens"`)?

  • DataBinding: View'et "abonnerer" på properties i ViewModel.
  • Observer Pattern i praksis: ViewModel implementerer et interface (ofte INotifyPropertyChanged). Når en variabel ændres i logikken, råber ViewModel "Hej, Navn er opdateret!".
  • View'et hører dette og opdaterer automatisk skærmen.

Del B: Hvordan ser det ud UDEN MVVM?

Før MVVM brugte man "Code-Behind" (Smart UI anti-mønsteret).

// MainWindow.xaml.cs (View og Logik er blandet sammen!)
public partial class MainWindow : Window {

    // Knap-klik er direkte bundet til UI'et
    private void btnGem_Click(object sender, RoutedEventArgs e) {
        // Tæt kobling til UI-elementer
        string kundenavn = txtNavn.Text;

        // Direkte databasekald i UI laget
        Database.Gem(kundenavn);

        lblStatus.Text = "Gemt!";
    }
}

Konsekvens: Det er umuligt at Unit Teste `btnGem_Click`, for testen vil forsøge at åbne et rigtigt grafisk vindue.

Del B: Implementering af ViewModel

Vi flytter logikken over i en ren C#-klasse. Den har ingen referencer til UI-komponenter.

// KundeViewModel.cs (Uddrag)
public class KundeViewModel : INotifyPropertyChanged {
    private string _navn;

    // En property som Viewet kan "Binde" til
    public string Navn {
        get => _navn;
        set {
            _navn = value;
            // Magien! Giver Viewet besked om at opdatere skærmen
            OnPropertyChanged(nameof(Navn));
        }
    }

    // Kommando i stedet for Click-event
    public ICommand GemCommand { get; }

    public KundeViewModel() {
        GemCommand = new RelayCommand(GemKunde);
    }
}

Del B: Implementering af View (XAML)

View'et består nu næsten kun af opsætning (markup). Al "Code-Behind" er slettet.

<!-- MainWindow.xaml -->
<Window.DataContext>
    <local:KundeViewModel /> <!-- Her sættes koblingen -->
</Window.DataContext>

<StackPanel>
    <!-- Databinding af tekst. "UpdateSourceTrigger" opdaterer live -->
    <TextBox Text="{Binding Navn, UpdateSourceTrigger=PropertyChanged}" />

    <!-- Tekstblokken reagerer automatisk på OnPropertyChanged -->
    <TextBlock Text="{Binding Navn}" />

    <!-- Binder til ICommand i stedet for at bruge Click="btn_Click" -->
    <Button Command="{Binding GemCommand}" Content="Gem Kunde" />
</StackPanel>

Hvorfor er dette så genialt til Test?

Problem før MVVM:

Hvis vi ville teste logikken bag en "Gem"-knap, skulle test-suiten klikke på en rigtig skærm. Det kaldes E2E (End-to-End) test og er utroligt langsomt og ustabilt.

Løsning med MVVM:

ViewModel er bare en almindelig C# klasse! I vores Unit Test kan vi bare skrive:

var vm = new KundeViewModel();
vm.Navn = "Jens";
vm.GemCommand.Execute(null);
Assert.True(db.ErGemt);

Den kritiske vinkel: Faldgruben ved MVVM

Boilerplate-kode og kompleksitet.

MVVM kræver utrolig meget opsætning (boilerplate). For blot at vise en tekst og trykke på en knap, skal man oprette implementeringer af `INotifyPropertyChanged`, `ICommand` og skrive bindings i XAML.

Til en lille, simpel applikation, hvor man bare vil beregne to tal, kan MVVM føles som over-engineering. Men i det øjeblik appen skal vedligeholdes af et team eller unit-testes, betaler det sig selv hjem tifoldigt.

Opsummering

  • Separation: Afkobler UI fra Forretningslogik. Designeren kan arbejde i XAML, mens koderen arbejder i C# ViewModel.
  • DataBinding: Er hjertet i MVVM. Ændringer flyder automatisk mellem UI og logik via Observer-mønstret (INotifyPropertyChanged).
  • Testbarhed: Fordi ViewModel ikke indeholder UI-elementer, kan alt, hvad brugeren gør, unit-testes lynhurtigt i isolation.
