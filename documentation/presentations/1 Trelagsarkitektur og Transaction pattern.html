Trelagsarkitektur (Separation of Concerns)

1. Præsentationslag (UI / API)
Viser data og tager imod input. Ingen logik!
↓
2. Logiklag (Business Logic)
Hjernen. Her bor vores Transaction Scripts.
↓
3. Datalag (Data Access Layer)
Gemmer og henter fra databasen via SQL/ORM.

Filosofien bag Transaction Script

I stedet for komplekse objekt-netværk, er logikken procedural.

  • En brugerhandling = Ét script (én metode).
  • Scriptet orkestrerer hele processen: Validering → Beregning → Gem → Returner svar.
  • Fordel: Lynhurtigt at udvikle, nemt at læse fra top til bund.

Løsningen i kode: Trelag + Script

// 1. UI Laget (Modtager anmodningen)
public IActionResult KøbVare(int vareId, int kundeId) {
    _ordreService.GennemførKøb(vareId, kundeId); // Kalder BLL
    return View("Succes");
}

// 2. Logik Laget (Transaction Script)
public void GennemførKøb(int vareId, int kundeId) {
    var vare = _db.HentVare(vareId);           // Spørg DAL
    if (vare.Lager == 0) throw new Exception(); // Validering

    decimal pris = vare.Pris * 1.25m;           // Beregning (Moms)

    _db.TrækPenge(kundeId, pris);               // Spørg DAL
    _db.OpdaterLager(vareId, -1);               // Spørg DAL
}

Hvordan ser det ud UDEN designet?

Anti-mønsteret: The Smart UI / Pærevælling.

// Alt ligger direkte i knap-klikket i brugergrænsefladen!
private void btnKøb_Click(object sender, EventArgs e) {
    // 1. Opret SQL Forbindelse i UI
    SqlConnection conn = new SqlConnection("Server=myServer...");

    // 2. Skriv rå logik og SQL sammen
    SqlCommand cmd = new SqlCommand("SELECT Lager FROM Varer...", conn);
    int lager = (int)cmd.ExecuteScalar();

    if (lager > 0) {
        decimal pris = int.Parse(txtPris.Text) * 1.25m;
        // Kør ny SQL for at opdatere...
        MessageBox.Show("Købt for " + pris);
    }
}

Konsekvensen af "Smart UI"

  • Umuligt at genbruge: Hvad hvis vi vil lave en mobil-app? Logikken er låst fast i hjemmesidens knap.
  • Sikkerhedsrisiko: SQL-forbindelser direkte fra præsentationslaget er farligt.
  • Test-mareridt: Hvordan skriver man en Unit Test for et knap-klik, der kræver en live database?

Testbarhed i Trelagsarkitektur

Ved at adskille lagene kan vi isolere vores logik i testfasen.

I vores test af GennemførKøb() transaction-scriptet, bytter vi det rigtige Datalag ud med et Mock Datalag.

  • Vi tester, at momsen regnes rigtigt, uden at røre en rigtig database.
  • Koden bliver robust, og tests kører på millisekunder.

Den kritiske vinkel: Faldgruben

Transaction Script er fantastisk til simple systemer, men fejler når kompleksiteten stiger.

Problemet:

Kode-duplikering. Hvis vi har 10 forskellige scripts, der skal beregne moms eller VIP-rabatter, ender vi med at kopiere/indsætte logik overalt. Det giver "Spaghetti-kode".

Løsningen:

Når systemet bliver for stort, skifter man typisk Transaction Script ud med en Domain Model, hvor logikken kapsles ind i intelligente objekter (f.eks. en `Ordre` klasse).

Opsummering

  • Trelagsarkitektur: Deler koden i UI, Logik og Data. Giver overblik og genbrugelighed.
  • Transaction Script: Simpel, procedural organisering af logik. Én handling = ét script.
  • Testbarhed: Lagdelingen tillader os at mocke databasen og teste forretningsreglerne isoleret.
