DAO & Unit of Work

Sikker datahåndtering og abstrakte databaser

Software Design og Test

Hvilket problem løser de to mønstre?

Problemet med Data Access

Når forretningslogik skriver direkte SQL eller bruger specifikke database-frameworks, bliver logikken bundet (tightly coupled) til den specifikke database.

Problemet med Transaktioner

Delvise opdateringer: Hvad sker der, hvis vi trækker penge fra en kunde, men databasen crasher lige inden, vi opdaterer lagerbeholdningen? Vi får ødelagt, inkonsekvent data.

Filosofien bag

"Abstraktion og 'Alt-eller-intet'."
  • DAO (Data Access Object / Repository): Giver en illusion om, at databasen blot er en in-memory samling af objekter (f.eks. en `List<User>`). Den skjuler SQL.
  • Unit of Work (UoW): Holder styr på alle de ændringer, du laver i systemet i løbet af en forretningshandling. Til sidst siger du "Gem", og den sikrer, at alt gemmes som én atomar transaktion (Alt lykkes, eller intet gemmes).

Løsningen i praksis (Designet)

De to mønstre arbejder næsten altid sammen.

Unit of Work er "chefen". Den ejer databaseforbindelsen og transaktionen. Den indeholder også en række DAO'er / Repositories, som udfører det specifikke arbejde for hver tabel (Kunder, Ordrer etc.).

Eksempel: Med Design (Det rene snit)

public void GennemførKøb(int kundeId, Ordre ordre) {
    // Vi bruger uow - vi kender IKKE til SQL eller EF Core her!
    var kunde = _uow.Kunder.Hent(kundeId); // DAO (Repository)

    kunde.TrækPenge(ordre.Total);
    _uow.Ordrer.Tilføj(ordre);           // DAO (Repository)

    // Udfører alt som én sikker transaktion
    _uow.Commit();                       // Unit of Work
}

Hvordan ser det ud UDEN designet?

Uden UoW udføres forespørgsler én ad gangen uden et sikkerhedsnet.

public void GennemførKøb(int kundeId, Ordre ordre) {
    // 1. Trækker pengene. Gemmes direkte i DB!
    _db.Execute("UPDATE Kunder SET Saldo = Saldo - ... ");

    // SERVER CRASH HER = Penge er trukket, men ingen ordre oprettet!

    // 2. Opretter ordren
    _db.Execute("INSERT INTO Ordrer ...");
}

Konsekvensen

Uden **Unit of Work** mister vi **Atomicity** (fra ACID-principperne i databaser). Vores data bliver inkonsistent ved uventede fejl.

Uden **DAO/Repository** bløder vores database-sprog (SQL, connection strings) ind i domænelogikken. Det gør systemet ufleksibelt; hvis vi vil skifte fra MySQL til MongoDB, skal vi omskrive hele kernesystemet.

Gør det koden mere testbar?

Dette er selve fundamentet for at teste forretningslogik!

  • Forretningslogikken (BLL) afhænger nu kun af et IUnitOfWork interface.
  • I vores Unit Tests kan vi indsætte en MockUnitOfWork.
  • Mock'en bruger falske in-memory lister i stedet for en rigtig database. Vi kan lynhurtigt teste: "Blev `Kunder.Opdater()` og `Ordrer.Tilføj()` kaldt før `Commit()`?" uden at røre et rigtigt filsystem.

Den kritiske vinkel: Faldgruben

"Dobbeltkonfekt" i moderne frameworks.

Hvis du bruger Entity Framework Core (C#) eller Hibernate (Java), implementerer de allerede disse mønstre indbygget!

I EF Core er DbContext faktisk en Unit of Work, og DbSet<T> er en DAO/Repository. At skrive sine egne UoW- og Repository-klasser udenom EF Core betragtes i dag af mange som "over-engineering", da man pakker et framework ind i et andet framework.

Opsummering

  • DAO / Repository: Skjuler databasens detaljer og lader os arbejde med data som in-memory objekter.
  • Unit of Work: Koordinerer ændringer og sikrer, at enten gemmes alt, eller også rulles alt tilbage (Transaktionssikkerhed).
  • Testbarhed: Interfacene gør det muligt fuldstændig at afkoble og mocke databasen under unit testing.
