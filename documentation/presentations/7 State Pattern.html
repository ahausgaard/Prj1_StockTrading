State Pattern

Sikker tilstandsstyring uden if/else-spaghetti

Software Design og Test

Del A: Hvad er formålet med mønstret?

"Lad objektet skifte adfærd, når dets interne tilstand ændres."

State Pattern er et adfærdsmønster. Det bruges til at bygge State Machines (Tilstandsmaskiner).

Formålet er at isolere den logik, der hører til en bestemt tilstand (f.eks. om et dokument er en "Kladde", "Til Godkendelse" eller "Udgivet"). Objektet ser ud til at skifte sin klasse undervejs i programmet.

Del B: Hvordan ser det ud UDEN mønstret?

Uden State Pattern ender klasser med komplekse livscyklusser med gigantiske, uoverskuelige switch-statements, der bryder Single Responsibility Principle.

public class Ordre {
    private string _tilstand = "Oprettet"; // "Betalt", "Afsendt"

    public void Betal() {
        if (_tilstand == "Oprettet") {
            // Godkend betaling
            _tilstand = "Betalt";
        }
        else if (_tilstand == "Betalt") {
            throw new Exception("Ordren er allerede betalt!");
        }
        else if (_tilstand == "Afsendt") {
            throw new Exception("Ordren er allerede afsendt!");
        }
    }
}

Del B: Implementering - Interfacet og Context

Vi starter med at definere Interfacet og give vores Ordre (Context) en tilstand, den kan ændre på.

// Interfacet: Definerer hvad man kan gøre med ordren
public interface IOrdreTilstand {
    void Betal(Ordre context);
    void Afsend(Ordre context);
}

// Context: Selve Ordren
public class Ordre {
    public IOrdreTilstand Tilstand { get; set; }

    public Ordre() {
        Tilstand = new NyOrdreTilstand(); // Start-tilstand
    }

    // Ordren delegerer arbejdet videre til sin nuværende tilstand!
    public void Betal() => Tilstand.Betal(this);
    public void Afsend() => Tilstand.Afsend(this);
}

Del B: Implementering - Konkrete Tilstande

Vi bryder nu if-statements op i separate, testbare klasser.

public class NyOrdreTilstand : IOrdreTilstand {
    public void Betal(Ordre context) {
        Console.WriteLine("Betaling modtaget.");
        // Tilstanden udskifter sig selv med den næste tilstand!
        context.Tilstand = new BetaltTilstand();
    }

    public void Afsend(Ordre context) {
        throw new Exception("Kan ikke afsende ubetalt ordre!");
    }
}

public class BetaltTilstand : IOrdreTilstand {
    public void Betal(Ordre context) => throw new Exception("Allerede betalt.");

    public void Afsend(Ordre context) {
        Console.WriteLine("Pakken er sendt afsted.");
        context.Tilstand = new AfsendtTilstand();
    }
}

Hvor og hvordan er det brugt? (Del B)

Anvendelsesscenarier

  • E-commerce Ordrer: (Ny → Betalt → Plukket → Sendt).
  • Spiludvikling (AI): En fjende-NPC skifter mellem IdleState, ChaseState, og AttackState afhængigt af afstanden til spilleren.
  • Hardware/Automater: En kaffemaskine, en hæveautomat eller en elevator.

Gør det koden mere testbar?

Ja! I stedet for at skrive én massiv Unit Test for `Ordre`-klassen, kan vi nu skrive små, isolerede tests for `NyOrdreTilstand` og bekræfte, at den kaster de rigtige exceptions og skifter til den korrekte næste tilstand.

Filosofien: Polymorfi frem for Betingelser

State-mønstret udnytter objektorienteret programmerings kernekoncept: Polymorfi (at objekter kan antage flere former).

I stedet for at koden spørger: "Hvilken tilstand er jeg i, og hvad skal jeg så gøre?", siger Context blot: "Gør dit job!" til den aktuelle tilstandsobjekt.

Det gør det ekstremt nemt at overholde Open/Closed Principle. Hvis vi får brug for en RefunderetTilstand, opretter vi bare en ny klasse – vi behøver ikke rette i den eksisterende kode.

Den kritiske vinkel: Faldgruben

Eksplosion af klasser og Over-engineering.

Hvis dit system kun har to tilstande (f.eks. "Tændt" / "Slukket"), og logikken for at skifte er meget simpel (f.eks. ændre en boolean), er State-mønstret "overkill". Det tvinger dig til at oprette adskillige klasser for at gøre noget, der kunne være løst med en simpel if-sætning.

Brug det kun, når tilstandene dikterer markant forskellig adfærd.

Opsummering

  • Mønstret: Erstatter komplekse if/else og switch-statements med dedikerede State-klasser.
  • State Machine: Context-objektet har en reference til den aktuelle IState, og State-klasserne orkestrerer selv overgangen til næste tilstand.
  • Vedligeholdelse: Ny adfærd tilføjes som nye klasser (Open/Closed), hvilket gør koden ren og isoleret.
