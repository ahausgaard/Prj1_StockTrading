Del A: V-Modellen & Del B: Teststrategier

Software Design og Test

Del A: Hvad er V-modellen?

"For hver udviklingsfase findes en tilsvarende testfase."

V-modellen er en udvidelse af vandfaldsmodellen.

I stedet for at udvikle alt først, og teste det hele til sidst, planlægger man tests parallelt med designet. Den venstre side (Design/Verifikation) spejler direkte den højre side (Test/Validering).

V-modellens Niveauer

Venstre: Design (Verifikation)

1. Kravspecifikation (Requirements)
2. System Arkitektur
3. Komponent Design
4. Modul / Logik Design
⟶

Højre: Test (Validering)

1. Accepttest (Gør systemet det kunden vil?)
2. Systemtest (Virker hele systemet samlet?)
3. Integrationstest (Snakker modulerne sammen?)
4. Unit Test (Virker den enkelte metode?)

I bunden af V'et finder vi selve KODNINGEN.

Del B: Unit Test vs. Integration Test

Unit Test (Enhedstest)

  • Tester den mindste del af koden (f.eks. én metode).
  • Total isolation: Ingen rigtige databaser, filer eller eksterne API'er.
  • Lynhurtig. Eksekverer på millisekunder.

Integration Test

  • Tester om to eller flere moduler fungerer sammen.
  • Interaktion: F.eks. Forretningslogik der snakker med en rigtig database via Entity Framework.
  • Langsommere, men fanger konfigurationsfejl.

Strategi for Unit Test: Grænseværdier

En af de stærkeste strategier til unit test kaldes Boundary Value Analysis (Grænseværdianalyse).

I stedet for at teste tilfældige tal, fokuserer vi på kanterne af de lovlige værdier (Equivalence Partitions), fordi det er her, programmører oftest laver fejl (f.eks. `<` i stedet for `<=`).

Eksempel: Adgang forbudt under 18 år.
Vi tester grænserne: 17 (skal fejle), 18 (skal lykkes), og 19 (skal lykkes).

Eksempel: Unit Test i kode (Boundary Value)

// Logikken vi tester
public bool IsEligibleForDiscount(int age) {
    return age >= 65; // Pensionistrabat
}

// Testen (XUnit syntax)
[Theory]
[InlineData(64, false)] // Lige under grænsen
[InlineData(65, true)]  // Lige på grænsen
[InlineData(66, true)]  // Lige over grænsen
public void Discount_CalculatesBoundaryValuesCorrectly(int age, bool expected) {
    // Arrange
    var calculator = new DiscountCalculator();

    // Act
    bool result = calculator.IsEligibleForDiscount(age);

    // Assert
    Assert.Equal(expected, result);
}

Strategi for Integration Test: Bottom-up

Når vi skal teste hvordan moduler snakker sammen, kan vi bruge en Bottom-Up (Nedefra-og-op) strategi.

  • Vi starter med at teste de underste lag i vores arkitektur (f.eks. Database Repositories).
  • Når vi er sikre på, at databasen virker, bygger vi Forretningslogikken (BLL) ovenpå og tester dem samlet.
  • Fordel: Vi fanger database/infrastruktur-fejl tidligt, før de forplanter sig til UI'et. Mocks er unødvendige for det nederste lag.

Eksempel: Integration Test i kode

Her tester vi Service-laget SAMMEN MED en ægte (ofte in-memory eller test-specifik) database.

[Fact]
public void GennemførKøb_TrækkerPengeFraRigtigDatabase() {
    // Arrange: Sæt rigtig DB op (In-Memory eller Test Container)
    var dbContext = new TestDbContext("IntegrationTestDB");
    dbContext.Kunder.Add(new Kunde { Id = 1, Saldo = 500 });
    dbContext.SaveChanges();

    // Inject rigtig DB i servicen
    var ordreService = new OrdreService(dbContext);

    // Act
    ordreService.GennemførKøb(kundeId: 1, pris: 200);

    // Assert: Tjek databasen for ændringer
    var opdateretKunde = dbContext.Kunder.Find(1);
    Assert.Equal(300, opdateretKunde.Saldo);
}

Den kritiske vinkel: Faldgruben

Overlapping Tests (Test-duplikering)

Mange teams ender med at teste nøjagtig det samme i deres Integration Tests som de allerede har gjort i deres Unit Tests. Dette fører til en utrolig langsom test-suite (ofte kaldet "The Ice Cream Cone anti-pattern").

Husk reglen: Unit tests tjekker al din komplekse forretningslogik (ifs, loops, math). Integration tests tjekker KUN, om "ledningerne" mellem systemerne er sat rigtigt sammen.

Opsummering

  • V-modellen: Forbinder krav og design på venstre side med konkrete testfaser på højre side.
  • Unit Test Strategi: Brug Boundary Value Analysis til effektivt at teste kerne-logikken isoleret.
  • Integration Test Strategi: Brug Bottom-up til at sikre, at dit Data Access lag og logiklag kan snakke fejlfrit sammen.
